## 2) Recommended minimal architecture (cost-aware)

## 2.0 Subdomain/service boundary decision (new)
- Use **`store.carapaceai.org`** for store/data APIs and internal Carapace store backend.
- Use **`ops.carapaceai.org`** for owner/admin operations.
- Use separate service boundaries even when temporarily co-hosted.

Security rationale:
- Clear trust boundary between owner-control ops plane and store/customer-facing data plane
- Easier policy isolation (auth scopes, rate limits, audit trails, WAF rules)
- Lower blast radius if one plane is degraded or compromised

Cost note:
- Additional subdomains themselves are effectively free in DNS.
- **Cost increases when we split to separate droplets/services**, not from DNS names.
- Current budget path: keep co-hosted initially if needed, then split to dedicated store droplet when store traffic/criticality rises.

## 2.1 Initial deployment (single VPS, segmented by service)
- VPS `carapace-core-01`
  - `updater-api`
  - `github-agent-runner`
  - `ops-cron-runner`
  - `ops-control-api` (single control plane endpoint for agent + admin UI)
  - reverse proxy + TLS

This is fast and cheap for now. Split to multiple droplets after workloads stabilize.

## 2.2 Planned split (security and blast-radius reduction)
1. **edge-updater** droplet
   - updater API + static installer artifacts
2. **ops-agent** droplet
   - GitHub manager OpenClaw agent + scheduled routines
3. **data-store** droplet
   - store metadata/data APIs + persistence
4. (future) **crypto-rt** droplet

## 3) Security model

## 3.1 Core controls
- SSH keys only (disable password auth after bootstrap)
- Per-service tokens (client token != admin token)
- Least-privilege credentials by service
- UFW/firewall deny-by-default
- Audit logs for admin actions

## 3.2 Secrets and rotation
- Store secrets in environment files with strict file perms
- Rotate API/admin tokens on schedule
- Rotate immediately after incident or suspected leak
- Never embed long-lived secrets in client binaries

## 3.3 Update feed controls
- Private endpoint with token auth
- Optional allowlist by tester/device later
- Signed artifact verification roadmap (next hardening step)

## 5.1 Ops Control API (agent-touchable server boundary)

Purpose: expose a narrow, authenticated API that both the GitHub agent and owner dashboard use, instead of allowing direct host/process control.

Initial endpoints:
- `GET /ops/health` - service and dependency status summary
- `GET /ops/services` - updater/agent/job service states
- `POST /ops/updater/publish` - publish metadata (and later upload trigger)
- `POST /ops/agent/github/command` - controlled commands (scan, triage, review sweep)
- `GET /ops/agent/github/status` - queue, last run, failures
- `POST /ops/jobs/run` - one-off safe routine execution
- `GET /ops/jobs/history` - scheduled run outcomes

Guardrails:
- role-based auth (owner/maintainer/viewer)
- command allowlist (no arbitrary shell)
- per-route audit logging
- rate limits and idempotency keys for mutations
